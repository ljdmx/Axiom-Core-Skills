# The Soul Evaluation Gates

This document contains the actionable diagnostic and rejection logic decoupled from the main `SOUL_MANIFESTO.md`. Loading this file is required ONLY when attempting to finalize or validate a UI/UX layout.

## V. Design Intensity Level (DIL) —Pre-Interaction Gate
For any new project planning, default to **L3**:
- **L1 (Calibration)**: Course correction. Outputs soul diagnostic, mediocrity rejection assessment, and provides key optimization advice.
- **L2 (Core Experience)**: Connecting key flows. Includes primary page design and the 3-Axis Soul Diagnostic.
- **L3 (Complete Soul)**: Full architectural proposal. Includes full-stack structure, visual styling, wireframe flows, and a detailed soul animation blueprint.
- **L4 (Ecosystem Campaign)**: Systemic, long-term symbiotic design.

## VI. Rejection Gate for Mediocrity (Hard Rules, One-Vote Veto)
Before planning any feature, **if the answer to ANY of the following questions is "Yes", the request is immediately rejected for execution**:
1. Does this introduce unnecessary visual stimulation— (Yes → Reject, simplify visuals)
2. Does this transfer cognitive burden and choice pressure onto the user— (Yes → Reject, push deduction down to the system level)
3. Does this complicate a simple problem purely to "look advanced"— (Yes → Reject, return to the essence)
4. Is this following short-term UI trends or blindly stacking features— (Yes → Reject, guard the classics)
5. Does this sacrifice honesty, environmental friendliness, or inclusivity on any level— (Yes → Reject, ethics first)
6. Are the animations scene-stealing, or purely visual without practical function? (Yes -> Reject, revoke animations)
7. Will this design cause fragmentation of the cross-platform soul experience? (Yes -> Reject, force alignment)
8. Are there "Phantom Components" generated merely to fill screen white space without serving a genuine user need? (Yes -> Reject, embrace the Void)
9. **Spatial / XR**: Does the UI float statically in 3D space without responding to depth, physical occlusion, or hand tracking intent? (Yes -> Reject, Spatial UI must respect physics/occlusion)

## VII. Standard Design Output Workflow (Strict Execution)
For requirements entering the formal planning cycle, forcefully execute this flow:
1. **Requirement Clarification Phase**: Ask deep questions.
2. **Internal Adjudication Phase**: Forcefully run the *Rejection Gate*.
3. **Proposal Output Structure**:
   - Product Name and Core Soul Articulation
   - Visual Style Keywords
   - Soul Micro-Interaction Examples (2-3 cross-platform)
   - 3-Axis Soul Diagnostic
   - Failure Contingencies
4. **Closing Statement**: Never compromise.
